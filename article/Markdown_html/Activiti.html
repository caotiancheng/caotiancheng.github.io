1 工作流引擎的相关简介

基于各种开源和企业级工作流引擎的设计，工作流引擎大致分为两个模块：执行引擎和设计器。设计器包含了对于BPMN2.0（工作流引擎所涉及的图元规范）相关图元逻辑模型到物理模型的存储，而执行引擎主要负责对业务人员所设计流程的分发，流程的权限映射，绑定表单的数据处理，以及在流程实际执行过程中根据具体情况做的自适应工作（这部分主要存在于企业级工作流引擎中）。

起初了解BPMN，了解Petri网，了解业务计算，是在学院张老师的课堂。BPMN，简单来讲，是一种规范，规范和标准的作用，可大可小。而业务流程管理，不是将业务流程用BPMN标准的图元画出大量的业务流程图在信息系统里展示出来。业务流程管理是动态的，是基于流程数据的。软件开发人员和业务人员各司其职，前者，负责BPMN图元的存储，逻辑模型设计，物理模型映射，如何绑定数据，如何将图元，也就是流程节点的任务分发给不同的工作组。而后者，则无须关心内部软件的逻辑，只负责采用系统的BPMN图元设计器，像在Process On或者PPT里面画流程图一样操作，赋予流程节点以任务，参与者，任务属性，节点自适应配置等的定义，交由引擎来处理，而与作图而后存储不同的是，这样的业务流程图元集合，第一，可以执行正确性校验，并被系统后台动态执行，第二，可以动态的绑定表单，绑定业务流程数据。第三，可以根据业务流程的随时变化，基于权限进行传阅，特事特办等等的流程变更执行操作。BPMN的魅力，就是用一种协议，达成了业务人员和软件人员在IT系统层面上的沟通，一端催生了支持流程节点复杂存储和复杂映射的引擎内核，另一端催生了千变万化的基于企业不同需求的业务流程。二者结合，相辅相成，经过巧妙的设计，构成了一个极具效用的BPM系统。

1.1 BPM PaaS的优势

BPM系统的前瞻性和优势在于：采用流程第一，部门第二的思想，进行数据和运营管理和IT技术实现，基于流程的节点传递和持续进行，通过IT手段消除了部门壁垒。举一个恰当也不恰当的例子，我们作为一个学生，当我们产生“借书”这一需求时，我们会去图书馆从进门找到书开始，去到服务台登记，服务台录入图书信息，在这之前查阅一下图书的预约信息，得到借书许可后，将信息录入图书管理信息系统，而我们拿到书离开。总体来看，我们实际上是走了一个借书的流程，在每个流程节点都产生了一些数据，最后我们借书的过程所导致的结果，便是在系统里生成了一个带有我们信息和图书信息的表单。

BPM系统正是看到了这样的本质，使得每个流程节点可以绑定表单数据，通过流程执行，表单数据被丰富，最后流程遇到结束事件，表单被存储。而这样处理的优势就在于：问题的高度抽象化，当我需要借书时，我构造一个借书流程绑定数据表，分发给不同工作组，最后由不同分工的人丰富这些数据表。当我需要办理银行卡时，构造一个办卡流程，办理信息更新服务时，同理……，在同一个平台通过构造不同流程，完成不同业务的执行，数据传递，这是一个轻量化且易于监控的平台。而采用传统的数据集成方法，我们的学生信息系统会集成财务，图书，成绩，选课等等不同的系统，系统造价高，底层数据同步，数据交叉，权限控制十分困难，牵一发而动全身，存在诸多不便，这便是BPM的优势所在，并在云服务的时代背景下，更加催生了各种部门协同，企业协同，云BPM，云ERP等等基于公有云，私有云的平台或者应用，AWS PaaS就是一个很好的范例。

1.2 BPM与其他企业系统的关系及其职能

而关于BPM系统，或者BPM PaaS平台和其他业务应用系统的关系，在这里做一个简单的介绍。Business Application System最核心的一个是：ERP（负责财务，销售，采购，库存，生产等等信息的集成），它所处理的是物流，信息流，资金流，三流合一，负责管理企业这些环节的生产数据：1. 生产控制（计划，制造）2. 物流管理（分销，采购，库存管理）3. 财务管理（会计核算，……）而其他常见的business system简列如下，例如：

CRM：客户关系管理

HRM：人力资源管理

OA：办公自动化（审批流程，知识管理，内部通知与内部即时通讯）

BPM：业务流程管理，集成度很高

WMS：仓库管理

BI：商务智能，数据整合，提供报表，提供决策依据

ERP和BPM的对比：ERP侧重的关键字：整合／集成 。例如生产部门的货供给销售部门信息，采购部门采购情况可以给生产部门信息。至于智能补货，或者智能调整销售策略和销售单价这些功能通过为企业定制化也可以辅助决策。而BPM侧重的关键字：流程。提升企业运营效率，各种流的监管和智能化，以达到高效化运营的目的。即BPM是战略和运营层的，ERP是作业层的。

BPM可以在理论上替代ERP，因为生产制造本质也是一种流程，但是基本不会有企业这样做，因为两种系统侧重点的不同以及代价问题所导致。在运行时，一般可以选择进销存数据接口和BPM系统的集成。以炎黄的系统举例，或许可以采用数据窗口的方式进行集成，这一点是我的小思考，不知准确与否。

而该BPM PaaS，所具备的职能如下图所示：流程资产库可以支持企业流程梳理，工作流引擎等可以支撑流程建模，BPA（Business Process Analysis）支持流程数据分析（财报销售如何，生产部门绩效如何，甚至请假流程的执行情况如何），Web端和移动端，支持移动办公，移动审批，并且基于AWS平台的AWS MVC框架，可以进行PaaS应用的开发和定制，进而产生众多云应用（IaaS关注基础设施管理，PaaS关注平台建设，而基于PaaS可以产生很多Software，即SaaS关注软件云服务，三种Service基于云，形成了一种类似于三层架构的关系），最终形成业务流程管理的生态系统，应用基于云，可卸载，可安装，可分发独立性和扩展性均十分具备优势。而对于技术设计架构，这里不加细节的赘述，仅做部署的简单介绍。而云BPM的高效职能，以及其在未来的发展中，除了在传统意义上Flash的停止支持带来Html 5 设计器的技术革新等等以外，是否能够将机器学习与深度学习，人工智能算法，流程知识感知算法，流程比对算法，流程预测等算法相结合，达到智能自动化流程申请审批（如语音识别与流程构造与执行相结合），智能风险控制，智能流程知识梳理、流程模型可配置与自适应等等效果，一切拭目以待。毕竟，这是一个伟大的时代。

2 Activiti源代码部署配置

2.1 Activiti源代码使用方法

2.1.1 源代码选择

Activiti同jBPM开源工具相同，具有eclipse插件。基于插件，开发人员可以进行流程应用开发，拖拽式流程设计和流程任务分发。除此之外，我们知道，activiti也有基于web端的拖拽式流程设计器，通过该设计器也可以进行流程节点任务的分发，表单映射等工作。而基于源代码的相关研究，Activiti和jBPM均分为几个大的研究脉络：

  1. 专注于引擎设计。针对引擎设计，可以直接在github进行引擎下载：https://github.com/Activiti/Activiti ，在该链接中develop分支或者master分支均只含有activiti 执行引擎的java代码，而不存在modeler（设计器）的部分，并且，内含各模块的jUnit测试类，开发人员可以使用java测试类进行流程的代码构造，进而测试不同的引擎模块。
  2. 专注于设计器和引擎设计。设计器部分即modeler的代码，如果希望了解两个大模块设计，建议下载：https://github.com/Activiti/Activiti/releases ，该链接地址含有6.0.0版本和5.22.0版本及其他版本的完整版源代码，本文做的解析包含设计器和引擎，因此采用的代码为5.22.0版本源代码，原因是该版本中存在package: activiti-explorer。这个模块含有web端的BPMN图元设计器，可以供开发人员或者业务人员（主要）进行业务流程的设计。所以本文通过这个设计器入口，也可以对图元从json到bpmn XML文件进行细致的分析。

无论是modeler，还是engine，在真正的企业化环境中，都有可以优化的很大空间，比如从用户体验、节点并发，大批量节点的数据传输，节点绑定表单的数据结构，流程执行的引擎执行算法，正确性校验算法等等，都需要巧妙和具有前瞻性的设计去支撑。因此，懂得工作流引擎的原理，对于在构造数字化企业的过程中，在选型和实施上都会带来极大帮助和启发。基于BPM engine core，在构造数字化平台生态过程中，平台应用的独立性，扩展性，集成难易度等等都是很值得去研究的领域，本文主要关注开源BPM engine core的设计，并结合AWS PaaS企业级BPM平台延伸一些相关的启发和心得。

2.2 Activiti源代码特点简述

2.2.1 Activiti引擎特点

Activiti的架构分为Modeling部分、Engine部分、Management部分。其中Modeling包括：modeler，desiginer，engine包括：Engine，Mangement包括：explorer，rest（提供restful服务，使客户端可以以json形式和引擎的rest API交互，跨平台，跨语言）。

  1. 开发人员了解，应用瓶颈在于和数据库的交换数据过程，Activiti采用myBatis，使其保持较高性能。
  2. 引擎Service简介：Activiti提供了7大流程接口，通过ProcessEngine获取。

ProcessEngines.getDefaultProcessEngine();这个方法被调用之后，会首先初始化并且创建一个流程引擎，并且以后会一直返回同一个流程引擎。ProcessEngine.init()和ProcessEngine.destroy方法会被用来创建和关闭所有流程引擎的属性。 ProcessEngine这个类会被所有的activiti.cfg.xml和activiti-context.xml文件扫描到。对于所有的activiti.cfg.xml来说，流程引擎都会用一种典型的方式被创建。这种方式就是：

 
ProcessEngineConfiguration.createProcessEngineConfigurationFromInputStream(inputStream).buildProcessEngine();

对于所有的activiti-context.xml来说，流程引擎会被用Spring的方式来创建。这种方式是：首先Spring的应用上下文被创建，然后流程定义会在应用的上下文中获得。

 
public interface EngineServices {
  RepositoryService getRepositoryService();
  RuntimeService getRuntimeService();
  FormService getFormService();
  TaskService getTaskService();
  HistoryService getHistoryService();
  IdentityService getIdentityService();
  ManagementService getManagementService();
  DynamicBpmnService getDynamicBpmnService();
  ProcessEngineConfiguration getProcessEngineConfiguration();
}

对于Service服务的不同职能请见下表：

Service nameFunction
RepositoryService流程仓库Service，用于管理流程仓库，部署，删除，读取流程资源
IdentifyService身份Service，可以管理和查询用户，组之间的关系
RuntimeService运行时Service，可以处理所有正在运行状态的流程实例，任务等
TaskService任务Service，用于管理，查询任务，例如，签收，办理，指派
FormService表单Service，用于读取和流程，任务相关的表单数据
HistoryService历史Service，可以查询所有历史数据，例如流程实例，任务，活动，变量，附件等
ManagementService引擎管理Service，和具体业务无关，查询引擎配置，数据库，作业等。

2.2.2 Activiti与jBPM对比

Activiti5使用Spring进行引擎配置以及各个Bean的管理，综合使用IoC和AOP技术，使用CXF作为Web Services实现的基础，使用MyBatis进行底层数据库ORM的管理，预先提供Bundle化包能较容易的与OSGi进行集成，通过与Mule ESB的集成和对外部服务（Web Service、RESTful等）的接口可以构建全面的SOA应用；jBPM5使用jBoss.org社区的大多数组件，以Drools Flow为核心组件作为流程引擎的核心构成，以Hibernate作为数据持久化ORM实现，采用基于JPA/JTA的可插拔的持久化和事务控制规范，使用Guvnor作为流程管理仓库，能够与Seam、Spring、OSGi等集成。

技术组成ActivitijBPM

设计模式Command模式、观察者模式等

从技术组成来看，Activiti最大的优势是采用了PVM（流程虚拟机），支持除了BPMN2.0规范之外的流程格式，与外部服务有良好的集成能力，延续了jBPM3、jBPM4良好的社区支持，服务接口清晰，链式API更为优雅；劣势是持久化层没有遵循JPA规范。

jBPM最大的优势是采用了Apache Mina异步通信技术，采用JPA/JTA持久化方面的标准，以功能齐全的Guvnor作为流程仓库，有RedHat的专业化支持；但其劣势也很明显，对自身技术依赖过紧且目前仅支持BPMN2。

3 Activiti数据库结构梳理

如2.1.2所言，源代码部署后，系统配置数据库脚本完成时，生成Activiti依赖的25张数据表，对于每张数据表的职能及其关系，该章节做简要介绍，ACT_RE_*: 'RE'表示repository。 这个前缀的表包含了流程定义和流程静态资源 （图片，规则等等）。ACT_RU_*: 'RU'表示runtime。 这些运行时的表，包含流程实例，任务，变量，异步任务，等运行中的数据。 Activiti只在流程实例执行过程中保存这些数据， 在流程结束时就会删除这些记录。 这样运行时表可以一直很小速度很快。ACT_ID_*: 'ID'表示identity。 这些表包含身份信息，比如用户组等。ACT_HI_*: 'HI'表示history。 这些表包含历史数据，比如历史流程实例， 变量，任务等等。ACT_GE_*: 通用数据， 用于不同场景下，如存放资源文件。具体每种分类由以下数据表组成：

  1. 资源库流程规则表

    1). act_re_deployment 部署信息表

    2). act_re_model 流程设计模型部署表

    3). act_re_procdef 流程定义数据表

  2. 运行时数据库表

    1). act_ru_execution 运行时流程执行实例表

    2). act_ru_identitylink 运行时流程人员表，主要存储任务节点与参与者的相关信息

    3). act_ru_task 运行时任务节点表

    4). act_ru_variable 运行时流程变量数据表

    5). act_ru_job 运行时定时任务数据表

4 Activiti Modeler逻辑梳理

简单的来说，bpmn-modeler主要负责将json数据转换为bpmn Model，再转换为bpmn文件进行存储，当系统启动时，可以考虑缓存机制将bpmn java model 加入缓存中，这样一来前端的获取速度就会加快，避免了数据库的操作。

4.1 BPMN图元设计存储逻辑

4.1.1 存储图元类型

首先，系统所处理的BPMN图元集合，即一个业务流程由以下几个部分组成：

  • 事件（Events）
  • 网关（Gateways）
  • 活动（Activities）

其中，事件指在一个流程的生命周期中发生了什么事情，或意外，或信号，或边界事件（类似于嵌入触发器）等等，而其他两个概念，顾名思义，具体各图元的规范，三个部分下面的图元细分及功能介绍，请参见BPMN2.0规范。

4.1.2 图元存储模块结构

存储结构的核心是不仅要考虑BPMN图元的存储，仍然要考虑数据的绑定，以及petri网的转化。因为petri网是有语义的，BPMN不具备语义，在执行层面一定是通过petri网作为中介进行，而在存储的语法层面只需要按图元元素进行分类即可。但存储时，要考虑各种task（task继承了activity，activity里面什么都没有，因为活动总是有各种任务的，但是各种任务可以统称为活动），各种gateway，各种Event，以及各种DataObject、Process、subProcess的共计89个图元bean（包含于package org.activiti.bpmn.model），当然，开发人员可以选择在这个目录下进行一些自定义图元的定义，来满足开发人员所需要的一些特定业务需求或者满足一些智能设计。比如，“业务建模引擎的自适应构建设计”这一科研项目中，所需要的可配置图元和自适应监控，或许可以在该目录进行定义。

在这个章节，以activity和task等不易区分的类的区别和联系做详细介绍，因为流程信息中，不同图元数据结构的处理较为纷繁复杂。

4.2.3 Expression

Activiti使用UEL处理表达式。表达式可用在很多场景下，比如Java服务任务，执行监听器，任务监听器和条件流。

ExpressionManager.createExpression()用以生成Expression

ProcessEngineConfigurationImpl.getExpressionManager()用以返回ExpressionManager

表达式的值（来自于用户手册）

  • Value expression：解析为值。默认，所有流程变量都可以使用。所有spring bean（spring环境中）也可以使用在表达式中。 一些实例：

     
    ${myVar}
    ${myBean.myProperty}

    ​

  • Method expression：调用一个方法，使用或不使用参数。当调用一个无参数的方法时，记得在方法名后添加空的括号（以区分值表达式）。 传递的参数可以是字符串也可以是表达式，它们会被自动解析。例子：

     
    ${printer.print()}
    ${myBean.addNewOrder('orderName')}
    ${myBean.doSomething(myVar, execution)}

    ​

注意这些表达式支持解析原始类型（包括比较），bean，list，数组和map。

在所有流程实例中，表达式中还可以使用一些默认对象：

  • execution：DelegateExecution提供外出执行的额外信息。
  • task：DelegateTask提供当前任务的额外信息。注意，只对任务监听器的表达式有效。
  • authenticatedUserId：当前登录的用户id。如果没有用户登录，这个变量就不可用。

而CommandContextIntercepter顾名思义，它是一个拦截器，拦截所有命令，在命令执行前后执行一些公共性操作。比如CommandContextIntercepter的核心方法execute：

 
public <T> T execute(Command<T> command) {    
CommandContext context = commandContextFactory.createCommandContext(command);  
try {  
  //执行前保存上下文  
  Context.setCommandContext(context);  
  Context.setProcessEngineConfiguration(processEngineConfiguration);  
  return next.execute(command);//执行命令  
    
} catch (Exception e) {  
  context.exception(e);  
} finally {  
  try {  
      //关闭上下文，内部会flush session，把数据持久化到db等  
    context.close();  
 } finally {  
     //释放上下文  
    Context.removeCommandContext();  
    Context.removeProcessEngineConfiguration();  
  }  
} 
return null;
}

6 Activiti PVM学习与实践

Process Virtal Machine,流程虚拟机API暴露了流程虚拟机的POJO核心，流程虚拟机API描述了一个工作流流程必备的组件，这些组件包括：

PvmProcessDefinition：流程的定义，形象点说就是用户画的那个图。静态含义。

PvmProcessInstance：流程实例，用户发起的某个PvmProcessDefinition的一个实例，动态含义。

PvmActivity：流程中的一个节点。

PvmTransition：衔接各个节点之间的路径，形象点说就是图中各个节点之间的连接线。

PvmEvent：流程执行过程中触发的事件。

以上这些组件很好地对一个流程进行了建模和抽象。每个组件都有很清晰的角色和职责划分。
